# 產品需求文件 (PRD) - 宮廟抽籤與捐獻系統

## 1. 文件資訊 (Document Info)
* **版本**：v2.0 (加入後端、紀錄與金流)
* **建立日期**：2026-04-22
* **狀態**：Review

## 2. 產品概述 (Product Overview)
* **背景說明**：為了進一步提升信眾的參與感、幫助人們回顧神明的指引，並維持系統營運，特在線上抽籤系統的基礎上，加入「抽籤紀錄」與「線上添香油錢」兩大核心機制。這使得專案從純靜態前端，升級為具備會員狀態與金流的完整全端應用。
* **目標**：
  - 建立使用者身分以永久保存個人求籤歷史。
  - 串接第三方金流，提供安全順暢的數位捐獻體驗。
* **定義與縮寫**：
  - **香油錢**：信眾對線上宮廟的隨喜捐獻 (Donation)。

## 3. 目標受眾與使用者情境 (Target Audience & Scenarios)
* **使用者畫像**：有信仰且需要人生指引的人、過去受過幫助想要還願或捐款維持網站營運的信眾。
* **使用者情境**：
  - **情境一 (查閱紀錄)**：小明幾個月前抽過一支籤，最近事情有了結果，他登入網站的「我的紀錄」頁面，重溫當時神明的題點。
  - **情境二 (添香油錢)**：小雅對於抽到的上上籤感到心情大好，點擊「添香油錢」，選擇 500 元並透過第三方支付完成付款，隨後收到了一張精美的線上感謝狀。

## 4. 範圍與邊界 (Scope)
* **核心範圍 (In Scope)**：
  - 前端抽籤與擲筊動畫互動介面。
  - 會員系統 (註冊、登入)。
  - 資料庫紀錄 (儲存使用者的抽籤結果、問題、時間)。
  - 金流串接 (支援線上刷卡或行動支付進行隨喜捐獻)。
* **不在範圍內 (Out of Scope)**：
  - 複雜的宮廟商城 (例如售賣法器、護身符)。
  - 實體點燈 / 安太歲等進階實體服務的串接。

## 5. 功能需求 (Functional Requirements)
* **[P0] 會員系統**：支援基本 Email 註冊登入（MVP 階段），未來可擴充第三方整合 (如 Google/Line 登入)。
* **[P0] 抽籤與擲筊互動**：使用者提問後進行擲筊與抽籤，產生結果後需由系統自動歸檔。
* **[P0] 個人抽籤紀錄 (History)**：
  - 列表顯示個人的所有過往抽籤 (要求包含日期、所問事項、籤號)。
  - 可展開查看該次詳細的籤詩與白話文解析。
* **[P0] 線上捐獻金流 (Donation)**：
  - 於導覽列與抽籤結果頁面提供「添香油錢」入口。
  - 系統提供預設金額選項 (如 100, 500, 1000) 以及自訂金額。
  - 在安全保護下呼叫第三方金流 API 結帳。
* **[P1] 捐獻感謝狀**：付款完成後，生成一張數位感謝狀，包含捐獻者姓名以資紀念。

## 6. 非功能需求 (Non-Functional Requirements)
* **效能要求**：支付狀態的回呼 (Webhook) 確認需快速並擁有容錯機制，避免遺漏訂單更新。
* **安全性**：
  - 密碼需進行強雜湊加密存放 (如 bcrypt)。
  - 系統絕不儲存信用卡等敏感付款資訊，一律由金流閘道器負責。
* **相容性**：因使用者主要可能來自社群分享，須完美支援手機的應用程式內建瀏覽器 (In-App Browser)。

## 7. 介面與使用者體驗 (UI/UX Requirement)
* **新增頁面**：
  - 「個人化會員中心 / 登入註冊」介面。
  - 「我的紀錄」歷史列表。
  - 「添香油錢」捐献金額選擇與結帳引導層。
* **設計準則**：維持宮廟特有的莊嚴古典、燙金質感風格。在結帳環節則須重視信任感與指示的清晰度。

## 8. 測試與驗收標準 (Acceptance Criteria)
* **資料庫驗證**：同一帳號能夠於不同裝置成功登入並查閱正確同步的歷史紀錄。
* **金流驗證**：使用測試信用卡號進行沙盒結帳測試，確認金流服務器回傳成功後，系統資料庫能正確產生一筆狀態為「已付款」的捐獻紀錄。

## 9. 數據指標 (Metrics / KPI)
* 註冊轉換率。
* 使用者復訪率 (Retention rate，衡量紀錄回顧功能的價值)。
* 捐獻成功率與平均客單價 (ARPA)。
